## 一、网络异常 — 全部通过 ✅

| 场景 | 状态 | 代码依据 |
|------|:---:|------|
| S01 WiFi→4G | ✅ | `ConnectionManager.NetworkMonitor` 监听 `onLost/onAvailable` → `scheduleReconnect("network_change")` → 指数退避 0s~300s |
| S02 短暂波动 | ✅ | WS `onFailure`/`onClosed` 触发重连；云中继 90s 心跳超时清理，重连后 PIN 重新映射 |
| S03 AP 漫游 | ✅ | TCP 断开被 WebSocket 层捕获 → `handleLanDisconnect()` / `handleCloudDisconnect()` → 自动重连 |
| S04 PC 断网 | ✅ | `isPcAlive()` 2s 超时(PING_TIMEOUT=2000ms) → fallback 云中继 `/api/v1/dial`；`PC_CACHE_TTL=30000ms` 过期后重新探测 |
| S05 PC 恢复 | ✅ | 30s 缓存过期 → 下次 `isPcAlive()` 重新 fetch → 可达 → 切回本地直连 |
| S06 云中继闪断 | ✅ | 优雅关闭: `for ws in ws_connections: await ws.close(1001)`；客户端 `onClosed` → 自动重连 |
| S07 手机无信号 | ✅ | 无信号期间 WS 无法收发 → 心跳超时(90s) → 云中继断开；信号恢复后`NetworkMonitor.onAvailable`触发重连 |
| S08 弱网络 | ✅ | WS 心跳包仅~50 bytes/30s，90s 超时容忍度足够；TCP 保证 dial 消息送达 |
| S09 VPN | ✅ | WS 路由基于 PIN 不依赖 IP，VPN 换 IP 不影响；除非 VPN 拦截端口 35430 |
| S10 基站切换 | ✅ | TCP 层透明，除非 IP 变化导致 TCP 重置 → `onFailure` 触发重连 |

## 四、号码类型 — 1 个潜在问题 ⚠️ 其余通过 ✅

| 场景 | 状态 | 代码依据 |
|------|:---:|------|
| S26 11位手机号 | ✅ | `1[3-9]\d{9}` 正则匹配 → `dial(number)` → 后端 `isValidDialNumber` 校验通过 |
| S27 10000 短号 | ✅ | 5 位纯数字 → 清理后 >=3 位 → 校验通过 → `telecomManager.placeCall(uri)` |
| S28 区号座机 | ✅ | `isValidDialNumber` 清除 `-`, ` `, `(`, `)` → "01012345678" 12位 → 3~20 范围 → 通过 |
| S29 400号码 | ✅ | 清理后 "4008008800" 10 位纯数字 → 通过 |
| S30 USSD(*100#) | ✅ | 清理 `*`, `#` → 检查剩余 "100" >=3 位 → 通过 → `Intent.ACTION_CALL` |
| S31 国际号码 | ✅ | 清理 `+` → "8613800138000" 13 位 → 通过 |
| S32 含空格 | ⚠️ | **后端清理但前端 content-script.js `getMyPhoneFromCRM()` 的 TreeWalker 返回的原始节点文本可能含空格** — `dial()` 传含空格号码到后端，后端能清理 ✅；但 `background.js` 中 `setPin` 设置 self_phone 时未清理空格 — 如果自动检测到的手机号前后有空格，将作为 PIN 存储 |

## 五、Chrome 扩展 — 2 个潜在问题 ⚠️

| 场景 | 状态 | 代码依据 |
|------|:---:|------|
| S33 多 Tab 多号码 | ✅ | 每个 Tab 独立 content-script.js → `chrome.runtime.sendMessage` → background.js 统一处理；云端 5s 去重仅同号码，不同号码并发通过 |
| S34 PIN 误匹配 | ⚠️ | **TreeWalker 扫描 body 全部文本节点**，取第一个 `1[3-9]\d{9}` 匹配 — 如果 CRM 页面订单号/身份证先于手机号出现，将误匹配；popup 中可手动修正但无自动纠正 |
| S35 Popup 打开拨号 | ⚠️ | **MV3 service worker 可能被 Chrome 在 30s 无活动后回收** — 如果用户先打开 popup 保持 30s+ 不动，再在 CRM 点号码，`background.js` 可能已被卸载 → `dial()` 无响应；实际概率低（popup 开着的场景较少） |
| S36 扩展更新 | ✅ | 更新后旧 content-script 卸载，需刷新 CRM 页面才能注入新版 — 这是 Chrome 机制，非代码问题 |
| S37 多 Profile | ✅ | 每个 Profile 独立的 `chrome.storage.local`，独立的 background.js service worker；同 PIN 下双方都建立独立的 pc_hello 连接 |
| S38 PIN 未设置 | ✅ | `getPin()` 返回 null → `dial()` 返回 "未检测到坐席手机号" → 不发送无效请求 |
| S39 服务器不可达 | ✅ | `fetch()` 超时 → catch → "网络错误，请检查云端服务器" → 不阻塞后续请求 |

## 六、Android 端 — 2 个潜在问题 ⚠️ 1 个缺失 ❌

| 场景 | 状态 | 代码依据 |
|------|:---:|------|
| S40 双卡 SIM 选择 | ✅ | `DialMode.POPUP`/`SYSTEM` → `Intent.ACTION_CALL` 系统弹窗；`DialAccessibilityService` 在非 POPUP 模式自动点击 |
| S41 SIM1 无信号 | ⚠️ | **`resolveSimSlot()` 不检查 SIM 是否可用** — 直接按模式返回 slot 0/1；如果 SIM1 无信号，`telecomManager.placeCall()` 仍尝试在 SIM1 拨出 → 系统可能提示失败或静默超时 |
| S42 无障碍未开启 | ✅ | `DialAccessibilityService` 需要用户在系统设置中开启；未开启时 `SIM1`/`SIM2` 模式下系统弹出 SIM 选择框无法自动点击 → App 界面提示"无障碍服务未开启" ✅ |
| S43 通话中收到 dial | ⚠️ | **`dialNumber()` 不检查当前通话状态** — 直接呼叫新号码；`endCall()` 方法存在但仅由 `hangup` 触发；连续 dial 可能在已有通话时失败 |
| S44 拨号后立即挂断 | ✅ | `hangup → endCall()` 调用 `TelecomManager.endCall()` → 取消呼出或挂断当前通话 |
| S45 存储不足 | ❌ | **`CallLogDb.insertDial()` 没有 try-catch 包裹 `writableDatabase.insert()`** — 磁盘满时 `SQLiteDiskIOException` 未捕获 → App 崩溃；对比：`updateSimCache()` 有 try-catch |
| S46 时间不同步 | ✅ | WebSocket 不校验时间戳；通话日志以 `System.currentTimeMillis()` 为准；功能不受影响 |
| S47 连续快速拨号 | ✅ | 5 个不同号码并发 `chrome.runtime.sendMessage` → background.js 依次 fetch → 云端 5s 去重不误触发；手机 `DialEngine` 逐号处理 |

## 七、并发与竞争 — 1 个潜在问题 ⚠️

| 场景 | 状态 | 代码依据 |
|------|:---:|------|
| S48 同 PIN 双手机 | ⚠️ | **如果 device_name 不同（如 "Redmi K40" vs "Redmi Note 12"），`phone_hello` 不会踢掉旧连接** → 两台手机同时在 `group.phones` 中 → dial 转发到两台 → 两台都拨出。若 device_name 相同则 close 旧连接 ✅ |
| S49 PIN 修改中拨号 | ✅ | 修改 PIN 不自动断开现有 WS（需要手动断开再重连）→ 旧 PIN 组正常转发；新 PIN 组下次重连生效 |
| S50 云中继重启瞬间 | ✅ | `shutdown()` → `stop_server()` → `for ws: close(1001)` → 客户端 `onClosed` 触发重连；重启期间 REST dial 返回 error（无 WS 服务）；重连后恢复 |
