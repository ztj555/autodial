## 一、全面审计结果

### 🔴 确认 Bug

#### B1. 场景 9/21：云中继连通 ≠ PC 在线

**代码位置**：`ConnectionManager.kt:812-819` (cloud `auth_ok` 处理)

```kotlin
"auth_ok" -> {
    cloudReconnectAttempts = 0
    manualConnecting = false
    transportMode = if (transportMode.contains("lan")) "lan+cloud" else "cloud"
    lastCloudPongTime = System.currentTimeMillis()
    cloudPingInFlight = false
    setState(ConnectionState.CONNECTED)  // ← 错误！立即设为 CONNECTED
}
```

**问题**：云中继服务器的 `auth_ok` 只代表手机↔云中继 WebSocket 通了，**不代表 PC 也连上了云中继**。PC 关机时云中继照样回 `auth_ok`。结果：手机显示绿色"已连接"、拨号指令发出去了，但云中继找不到 PC → 指令静默丢失。用户不知道为什么拨不了号。

**影响范围**：
- `ConnectFragment` 顶部状态显示绿色 + "已连接"
- `isConnected` 返回 `true`（因为 `isCloudConnected` 为 `true`）
- PC 端点拨号 → 手机收不到 dial 消息 → 无反应

---

#### B2. `isConnected` 语义被污染

**代码位置**：`ConnectionManager.kt:149-153`

```kotlin
val isConnected: Boolean
    get() {
        val lanOk = lanWebSocket != null && transportMode.contains("lan")
        val cloudOk = cloudWebSocket != null && transportMode.contains("cloud")
        return lanOk || cloudOk
    }
```

**问题**：`isConnected` 用来判断"是否已连接"，但云通道的 `cloudOk` 只是 WebSocket 存活，不验证 PC 可达性。导致所有依赖 `isConnected` 的逻辑（连接页 UI、按钮状态、通知更新等）都可能在 PC 离线时显示"已连接"。

---

#### B3. `performDial` 中 `startActivity` fallback 仍有静默失败风险

**代码位置**：`DialEngine.kt:180-188`

```kotlin
// fallback: ACTION_CALL
try {
    val intent = Intent(Intent.ACTION_CALL).apply {
        data = uri
        addFlags(Intent.FLAG_ACTIVITY_NEW_TASK)
    }
    service.startActivity(intent)
    Log.d(TAG, "dialed(ACTION_CALL fallback): $number")
    onDialSuccess(number, simSlot)  // ← startActivity 被静默拦截也不会抛异常
```

**问题**：v4.45 已经把 `placeCall` 提到第一位，大部分设备不会走到这个 fallback。但如果在某些设备上 `placeCall` 抛了异常，或 handle 为空（未识别到 SIM），就会走 `startActivity`。后台时被静默拦截，`onDialSuccess` 仍然执行 → 动画显示、记录写"ok"，实际没拨出去。

**建议**：`startActivity` 后加一个延迟检查（注册短暂的通话状态监听），如果 3 秒内没进入 OFFHOOK，将记录修正为 "error"。

---

#### B4. `manual_disconnect` 清除时机不完整

**代码位置**：`MainActivity.kt:77-78` + `ConnectionManager.kt:306-307`

```kotlin
// MainActivity.onCreate
getSharedPreferences("autodial", MODE_PRIVATE).edit()
    .putBoolean("manual_disconnect", false).apply()

// ConnectionManager.loadSavedConfig
if (prefs.getBoolean("manual_disconnect", false)) autoReconnect = false
```

**问题**：`manual_disconnect` 标志只在 `MainActivity.onCreate` 中清除。如果 `DialService` 在后台存活期间被系统重启（Android 杀掉进程后重建），`onCreate` 不会执行 `MainActivity.onCreate`（因为 Activity 还没创建），但 `DialService.onCreate` 会调用 `loadSavedConfig()` → `autoReconnect = false` → 服务启动后不重连。

**建议**：在 `DialService.onCreate` 中也清除 `manual_disconnect`，或者改方案——用内存标志而非 SharedPreferences。

### 🟡 潜在问题（不算 Bug 但值得改进）

#### P1. 退出连接页时 `connecting` 标志没复位

**代码位置**：`ConnectFragment.kt`

`connecting` 是 Fragment 的局部变量。如果用户连接过程中切到记录页（ViewPager 切换），Fragment 的 View 被销毁，但 `connecting` 标志还在。用户切回来时可能按钮状态不对。

#### P2. `ConnectFragment` 中 `connectionReceiver` 空实现

**代码位置**：`MainActivity.kt:42-44`

```kotlin
private val connectionReceiver = object : BroadcastReceiver() {
    override fun onReceive(context: Context?, intent: Intent?) {
        // 连接成功不再自动跳转到记录页
    }
}
```

确认了这是有意为之（之前已修复"连接成功不跳转"）。但空实现的 receiver 仍然消耗系统资源，可以考虑在注册后如果确定不再需要就取消注册。

#### P3. `DialEngine.dialNumber` 缺少 `simSlot == -2` 路径的后台保护

**代码位置**：`DialEngine.kt:125-131`

```kotlin
if (simSlot == -2) {
    val intent = Intent(Intent.ACTION_CALL).apply { ... }
    service.startActivity(intent)
    onDialSuccess(number, -1)
    return
}
```

系统拨号模式直接用 `startActivity`，在后台 100% 被拦截。但用户选"系统"模式时，本身就期望系统拨号器弹出来，所以可以接受。

## 二、场景 9/21 修复方案

### 方案设计

**核心思路**：云中继 `auth_ok` 后，不立即设 `CONNECTED`，而是发一个 `pc_hello` 消息，等待 PC 通过云中继回复。只有收到 PC 的回复，才确认 PC 在线。

### 协议改动（手机 ↔ 云中继 ↔ PC）

```
现有流程:
手机 ──phone_hello──→ 云中继 ──phone_hello──→ PC
手机 ←──auth_ok────── 云中继 ←──auth_ok──────── PC
                                        ↑ 云中继自己的 auth_ok 也行！

改进后:
手机 ──phone_hello──→ 云中继 ──phone_hello──→ PC
手机 ←──auth_ok────── 云中继 ←──auth_ok──────── PC
                                    ↓ (auth_ok 里附带 pong 或新增 pc_present 字段)
手机 收到 auth_ok 里有 pc_present: true → CONNECTED
手机 收到 auth_ok 里有 pc_present: false → 只设 relay_connected，不设 CONNECTED
```

**如果不想改云中继**，纯手机端方案：

```
手机 ←──auth_ok────── 云中继 (不管PC在不在)
手机 发送 ping 给云中继
  ├─ PC在线 → pong 通过云中继回来 → CONNECTED
  └─ PC不在 → 超时 → relay_connected, PC_OFFLINE
```

### 纯手机端实现步骤

**1. ConnectionManager 新增状态**

```kotlin
// 新增：云中继连通但PC状态未知时的中间状态
private var pcConfirmedOnline = false
val isPcReachable: Boolean get() = isLanConnected || (isCloudConnected && pcConfirmedOnline)
```

**2. 修改 cloud auth_ok 处理**

```kotlin
"auth_ok" -> {
    cloudReconnectAttempts = 0
    manualConnecting = false
    transportMode = if (transportMode.contains("lan")) "lan+cloud" else "cloud"
    lastCloudPongTime = System.currentTimeMillis()
    cloudPingInFlight = false

    // 不发 CONNECTED，先确认 PC 可达
    if (pcConfirmedOnline) {
        setState(ConnectionState.CONNECTED)
    } else {
        // 通过云中继 ping PC，超时 5s
        verifyPcReachable()
    }
}
```

**3. `verifyPcReachable` 方法**

```kotlin
private fun verifyPcReachable() {
    val ws = cloudWebSocket ?: return
    try {
        ws.send(JSONObject().apply { put("type", "ping") }.toString())
    } catch (_: Exception) {}

    handler.postDelayed({
        if (!pcConfirmedOnline) {
            // 5s 内没收到 pong → PC 不在线
            pcConfirmedOnline = false
            setState(ConnectionState.CONNECTED) // 仍然是 CONNECTED，但连接页显示不同
            notifyPcOffline()
        }
    }, 5000)
}
```

**4. 修改 pong 处理（收到即确认 PC 在线）**

```kotlin
"pong" -> {
    pcConfirmedOnline = true  // 云通道的 pong 是 PC 回过来的
    lastCloudPongTime = System.currentTimeMillis()
    cloudPingInFlight = false
    cloudLatencyMs = lastCloudPongTime - cloudPingSentTime
    if (state != ConnectionState.CONNECTED) {
        setState(ConnectionState.CONNECTED)
    }
}
```

**5. ConnectFragment 状态显示**

```kotlin
val cloudOk = DialService.isCloudConnected
val pcOk = DialService._instance?.connectionManager?.isPcReachable ?: false
cloudStatusText.text = when {
    cloudOk && pcOk -> "已连接"
    cloudOk && !pcOk -> "PC未上线"
    else -> "未连接"
}
```

### 影响评估

| 改动文件 | 行数 | 风险 |
|----------|------|------|
| `ConnectionManager.kt` | ~30 行 | 低 — 新增状态变量，不改核心流程 |
| `ConnectFragment.kt` | ~5 行 | 低 — 只改状态文字 |
| `DialService.kt`（暴露 `isPcReachable`） | ~3 行 | 极低 |

**不涉及云中继/PC端改动**，纯手机端闭环。

---

### 备选方案（更轻量）

如果不想加 ping/pong 验证，可以简化：**任何从云通道收到的业务消息（dial/sms等）都视为 PC 在线确认**。首次连接后不发 ping，等第一次拨号指令到达时自动确认。缺点：用户看到"PC未上线"直到第一次拨号前。
