## 二、连接路径矩阵

### 场景一：全本地（PC + 手机在同一网络）

```
插件端 → HTTP 35432 → PC端 → 直连 → 手机端
                              不需要云端 ✅
```

### 场景二：公网新云端

```
插件端 → REST 35441 → v3云 → WS 35440 → 手机端(11位JWT)
                               同一台服务器 ✅
```

### 场景三：混合（手机在老云端，插件在新云端）

```
插件端 → REST 35441 → v3云 → ??? → 手机端(老云端PIN)
                         ❌ v3云和老云无法互通
```

### 场景四：插件和手机不在同一网络（异地）

```
插件端 → REST 35441 → v3公网云 → WS 35440 → 手机端(需连公网云)
                                       ✅ 手机必须连云，不能只连LAN
```

> 如果手机只连本地 LAN PC（没连公网云），插件异地拨号无法触达。此时只能走 PC 直连——但插件和 PC 必须在同一网络。

## 三、兼容方案

### 3.1 手机端输入规则（已实现）

```
输入 4 位 → 老版本 PIN → WS 老云端
输入 11 位 → 新版本 JWT → auto-login → WS 新云端
```

**✅ 已经实现，无需改动。**

### 3.2 手机端需要改进

| 项目 | 当前状态 | 建议 |
|------|---------|------|
| 服务器列表 | 默认 `vicp.fun`（老云） | 首次使用引导添加新云地址 |
| JWT auto-login URL | 需从 WS URL 推导 REST URL | `ws://IP:35440` → `http://IP:35441/api/v1/auth/auto-login`，已实现 ✅ |
| 连接方式提示 | 无 | 输入框 placeholder 改为"4位配对码 或 11位手机号" |

### 3.3 插件端需要改进

| 项目 | 当前状态 | 建议 |
|------|---------|------|
| 云服务器地址 | 默认 `127.0.0.1:35441` | 本地用默认值；部署公网时需要弹出设置引导 |
| 错误提示 | "服务器不可达 (端口35441)" | 本地够用，公网加超时保护（见 CRM技术分析.md） |
| PC 检测 | 页面加载时检测一次，缓存 | 可以增加手动刷新入口（右键菜单"刷新PC状态"） |

### 3.4 PC端

> 改造计划书标记为"过渡期保留（不动）"。只连局域网和老云端。

## 四、风险和不合理之处

### ⚠️ 风险一：插件永远到不了老云端
- 插件只支持 REST API (35441)，老云端只有 WebSocket PIN
- **影响**：手机在老云上的 iOS/Android 用户，插件端无法给他们拨号
- **缓解**：推动手机端切换到新云

### ⚠️ 风险二：手机 AUTO 模式同时连 LAN + Cloud，拨号走哪条？
- 手机同时连 LAN(PC) 和云时，**手机端**会同时收到两条路径的消息
- 但这条路径的主动权在**发起端**：
  - 插件发起拨号时：检测到 PC 在线 → 走 PC 直连 → `return`，**不会**再走云
  - 所以**插件端不会重复发拨号指令**
- 真正需要关注的是：如果手机同时在 LAN 和云上，哪条路径会实际触达手机？
  - PC 在线：插件 → PC(HTTP) → PC 通过 LAN/云发给手机 ✅
  - PC 不在线：插件 → v3云 → 手机(JWT) ✅
- **结论：不存在重复拨号，插件端的 dial() 逻辑已做好互斥。**

### ⚠️ 风险三：PC端和插件端不一致
- 插件→PC(HTTP)→手机 拨号成功
- 但插件不知道手机是否在线（isPcAlive 只检测 PC）
- PC 转发拨号指令给手机，但手机可能关机/断连 → PC 返回什么？插件如何处理？
- **现状**：PC 直连 dial 接口无详细错误返回，失败时插件走云端兜底

### ⚠️ 风险四：老云端和新云端的认证隔离
- 同一个手机号，在老云端是 PIN 1234，在新云端是 JWT
- 两套认证体系互不相通
- **不会冲突**，因为走不同的服务器地址

### ⚠️ 风险五：PC 能否连 v3 云中继？

**可以，只需改 PC 端的服务器地址。** v3 云中继支持 PIN 认证：

```python
# cloud_relay_v3.py 实际代码
if auth_method == "pin":
    pin = msg.get("pin", "")
    group = get_group(pin)  # 注册到 PIN 分组
```

**不会冲突**，因为 v3 内部 PIN 分组和 JWT 设备是两套独立数据结构：
- PIN 消息走 `pin_groups["1234"]` → 老派路由
- JWT 消息走 `jwt_devices[user_id]` → 新派路由

**但有一个注意点**：插件端通过 REST API 发起的拨号，v3 只会转发给同 `user_id` 的 JWT 设备，不会转发给 PIN 设备。也就是说 **PC 连 v3 用 PIN 时，插件发起的拨号到达不了这台 PC**。不过 PC 如果和手机在同一个 PIN 组，手机直接拨号就行，不经过插件。

**建议**：PC 端连 v3 时可以保留 PIN，但标注"此模式下插件端无法远程拨号"。

### ⚠️ 风险六：公网部署时 fetch 无超时

> 详见 `AutoDial-Extension/CRM技术分析.md` 末尾表格。

- localhost (127.0.0.1)：端口关闭时 TCP 瞬间拒绝 → **无影响**
- 公网（腾讯云）：服务器宕机/网络断时，`autoLogin()` 和 `dial()` 的 fetch 死等 **30-120 秒**
- **当前无需处理**（本地部署不受影响）
- **部署公网前必须处理**：给这两个 fetch 加 5s AbortController

## 五、推荐部署架构

```
公网服务器 (腾讯云)
├─ cloud-relay-v3.exe
│   ├─ WS:35440  ← 手机端连接
│   └─ REST:35441 ← 插件端连接
│
本地电脑
├─ PC端:35432（可选）
└─ 插件端 → 公网 35441 或 本地 35432

手机
└─ 连接 公网 WS 35440 或 本地 LAN PC
```
