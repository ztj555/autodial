# 系统架构与配置说明 v3

> 基于实际代码 | 双版本共存

## 架构演进

```
v2 (旧):  CRM → 浏览器插件 → PC端(35432) → 手机
                                └→ 云中继(35430) → 手机（跨网）

v3 (新):  CRM → 浏览器插件 → 云中继v3(35441 REST) → 云中继v3(35440 WS) → 手机
              ↑ JWT认证                              ↑ JWT握手

过渡期:   CRM → 浏览器插件 → 优先PC(35432) → 手机
                          └→ 备选云中继v3(35441) → 手机
```

## 端口体系

| 端口 | 协议 | 版本 | 认证 | 用途 |
|------|------|------|------|------|
| 35440 | WS | v3 | JWT | 云中继 v3 WebSocket 中继 |
| 35441 | HTTP | v3 | JWT (Bearer) | 云中继 v3 REST API |
| 35430 | WS+HTTP | v2 | PIN | 云中继 v2 + Web 管理 |
| 35432 | HTTP+WS | v2 | 无/PIN | PC 端主服务 |
| 35433 | UDP | v2 | 无 | LAN 设备发现 |

> 新老端口分离，两套同时运行，互不干扰。

## v3 核心特性

### 安全
- bcrypt 密码哈希（原文不落地）
- JWT 短有效期（15min）+ refresh_token（30天，可吊销）
- IP + 手机号双维度防爆破（5次/15min）
- JWT 密钥三层保护：环境变量 > 0600 文件 > 自动生成

### 认证流
```
注册/登录 → bcrypt → 返回 JWT(15min) + refresh_token(30天)
自动续期 → /api/v1/auth/refresh → 新JWT + 新refresh_token
强制下线 → UPDATE refresh_tokens SET revoked=1
```

### 拨号确认
```
POST /api/v1/dial → 202 {req_id, status:"pending"}
手机拨出 → WS dial_ack → 更新结果
GET /api/v1/dial/result?req_id=xxx → {status:"ok"}
```

## 部署建议

1. **纯 v3 部署**：只启动 cloud_relay_v3.py（35440/35441），扩展和手机都走 JWT
2. **过渡部署**：同时启动 v2+v3，老用户继续用 PIN，新用户用 JWT
3. **开发测试**：所有服务都启动，端口不冲突

## 防火墙要求

- v3: 35440 (TCP), 35441 (TCP)
- v2: 35430 (TCP), 35432 (TCP), 35433 (UDP)
