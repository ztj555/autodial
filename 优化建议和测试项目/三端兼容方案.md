## 二、连接路径矩阵

### 场景一：全本地（PC + 手机在同一网络）

```
扩展 → HTTP localhost:35432 → PC端 → WS 直连 → 手机
                                    不需要云端 ✅
```

最佳体验：延迟 < 10ms，零依赖外部服务。

### 场景二：异地（扩展和手机不在同一网络）

```
扩展 → REST 35430 → 云中继 → WS 35430 → 手机
        ↑ X-AutoDial-PIN         ↑ PIN 认证
```

扩展通过公网云中继拨号，手机需连接同一云中继。

### 场景三：混合通道（PC 连云，扩展走本地）

```
扩展 → HTTP localhost:35432 → PC端 → WS 35430（云中继）→ 手机
```

PC 作为桥梁，扩展无需知道手机的具体位置。这是生产中最灵活的模式。

### 场景四：Go PC 替代 Electron PC

```
扩展 → HTTP localhost:35432 → Go PC端 → WS → 手机
                              ↑ 协议完全兼容
```

扩展无法区分是 Electron 版还是 Go 版——通信协议完全相同。

## 三、三端连接能力

| 连接路径 | PC (Electron) | PC (Go) | 扩展 | 手机 |
|----------|:---:|:---:|:---:|:---:|
| 局域网直连 (WS 35432) | ✅ | ✅ | ✅ (HTTP) | ✅ |
| 云中继 v2 (WS 35430) | ✅ | ✅ | ✅ (REST) | ✅ |
| 云中继 v3 JWT (WS 35440) | ❌ | ❌ | ❌ | ✅ |

> v3 JWT 中继仅手机端支持（Android PIN+JWT 双兼容）。PC 端和扩展均未集成 JWT 认证。

## 四、PC 端双实现影响

### 兼容性

- **手机端**：透明——与哪个 PC 实现通信完全无感
- **扩展端**：透明——HTTP 35432 接口完全一致
- **云中继**：透明——`pc_hello` 消息格式完全相同

### 差异影响

| 差异点 | 影响 |
|--------|------|
| 连接上限：Electron 2台 vs Go 10台 | 多手机场景使用 Go 版 |
| 窗口数：Electron 4窗 vs Go 1窗 | Go 版界面更紧凑 |
| 短信窗口：Electron 有 vs Go 无 | Electron 版短信功能更完整 |
| 安装体积：Electron ~150MB vs Go ~10MB | Go 版分发更轻量 |

## 五、v3 JWT 模块处理策略

### 当前状态

`cloud_relay_v3.py` + `auth.py` + `db.py` 在代码库中保持完整可用，运行在独立端口 35440/35441。

### 是否应删除？

**不建议**。理由：
1. Android v4.53 手机端仍支持 JWT 握手（`auth_method: "jwt"`）
2. 如果某 Android 用户仍使用 v3 云中继配置，删代码会导致该用户无法连接
3. 独立端口隔离，不影响主中继运行

### 建议

| 方案 | 说明 |
|------|------|
| **短期** | 保留代码，标注注释明确 v3 为并存模块、非主用 |
| **中期** | 在 Android 端移除 JWT 握手代码，待所有用户迁移后再删除 v3 代码 |
| **长期** | 确认无用户依赖后统一删除 v3 模块和 `requirements.txt` 中增量依赖 |

## 六、潜在风险

| # | 风险 | 严重度 | 现状 |
|---|------|--------|------|
| 1 | 手机同时连 LAN + Cloud，拨号可能双路到达 | 低 | 扩展 `dial()` 已做互斥（PC 在线时不走云），手机端 ACK 机制去重 |
| 2 | PC 直连拨号失败时错误信息不具体 | 低 | PC 返回通用错误，扩展走云端兜底 |
| 3 | 云端 fetch 无超时（公网部署时） | 中 | localhost 不受影响；公网部署前需加 AbortController 5s 超时 |
| 4 | v3 JWT 代码保留可能导致新开发者误解 | 低 | 需清晰注释声明为主中继并存模块 |
| 5 | Go 版 `sendToPhone` 遍历所有设备 | 中 | 多设备场景下拨号可能发给非目标设备（Electron 版只发活跃设备） |
| 6 | requirements.txt 含 v3 增量依赖 | 低 | 主中继部署时这些依赖安装后不会被使用，体积稍大 |
