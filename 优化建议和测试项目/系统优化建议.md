## 一、架构现状

```
┌──────────┐    ┌──────────────┐    ┌───────────────────┐
│ CRM网页   │───→│ Chrome扩展    │───→│ PC端 (二选一)       │
│          │    │ v4.0.0       │    │ Electron v3.0.0   │
│          │    │ MV3 SW       │    │ 或 Go/Wails v1.0  │
└──────────┘    └──────┬───────┘    └────────┬──────────┘
                       │                     │
               优先: HTTP 35432       LAN: WS 35432
               降级: REST 35430     Cloud: WS 35430
                       │                     │
                       └─────────┬───────────┘
                                 │
                       ┌─────────▼──────────┐
                       │   云中继 v2 (35430) │
                       │   cloud_relay.py   │
                       │   PinGroup 分组     │
                       │   (零数据库依赖)    │
                       └─────────┬──────────┘
                                 │
                       ┌─────────▼──────────┐
                       │  Android 手机端     │
                       │  v4.53             │
                       │  PIN + JWT 双兼容  │
                       └────────────────────┘
```

**端口体系**：

| 端口 | 协议 | 认证 | 用途 |
|------|------|------|------|
| **35430** | WS + HTTP | 11 位 PIN | 云中继主端口（WS 中继 + REST API + Web 管理） |
| 35431 | HTTP | 无 | Web 管理面板（dashboard） |
| 35432 | HTTP + WS | PIN（LAN）/ 无（localhost） | PC 端主服务 |
| 35433 | UDP | 无 | LAN 设备发现 |
| 35440 | WS | JWT + PIN | v3 JWT 云中继（并存模块，非主用） |
| 35441 | HTTP | JWT | v3 JWT REST API（并存模块，非主用） |

## 二、核心特性

### 安全设计

| 机制 | 说明 |
|------|------|
| **PIN 强校验** | 全链路 11 位数字校验（PC 端 + 云中继 + 扩展） |
| **并发保护** | PC_CONNECTED 去重 + DUPLICATE_DIAL 5s 去重 |
| **频率限制** | 每 IP 每分钟 5 次握手尝试 |
| **心跳超时** | WebSocket ping/pong（云端 30s/90s，PC 端 15s/20s） |
| **空 PIN 守卫** | PC 端未设置 PIN 时拒绝一切连接 |
| **generation 防竞态** | PC 端云中转使用递增 generation 号防止旧连接事件覆盖 |
| **ACK 确认** | 拨号指令 3s ACK 超时，自动切通道重试 |
| **零数据库** | 云中继主版本无数据库依赖 |

### 连接管理

**双通道设计**（Electron PC 端 PhoneConnectionManager / Android ConnectionManager）：
- LAN 优先（WebSocket 直连，延迟 < 10ms）
- Cloud 降级（WebSocket 中继，跨网使用）
- 自动切换，对用户透明

**拨号队列**：手机离线时自动暂存拨号请求（30s 超时），重连后自动补发。

### 认证流

```
CRM 自动检测坐席手机号 → 存为 PIN → X-AutoDial-PIN Header → 云端 11 位校验
                                                             ├─ 不通过 → INVALID_PIN
                                                             └─ 通过 → 查 PinGroup → 转发
```

## 三、PC 端双实现演进

| 特征 | Electron v3.0.0 | Go/Wails v1.0.0 |
|------|----------------|-----------------|
| 运行时依赖 | Electron (~150MB) | 单文件 exe (~10MB) |
| 代码架构 | 10 模块 + main.js (919行) 编排 | 9 个 .go 文件按职责拆分 |
| 窗口数 | 4 个独立窗口 | 1 个（内嵌） |
| 连接上限 | 10 台手机 | 10 台手机 |
| 通信协议 | **相同** | **相同** |

> 两版可互换使用。Go 版是明确的演进方向（更轻量、原生 API），Electron 版功能更完整（短信窗口、多主题、文件上传桩）。

## 四、v3 JWT 模块的并存状态

代码库中 **仍保留** 完整的 v3 JWT 中继模块（`cloud_relay_v3.py` + `auth.py` + `db.py`），运行在独立端口 35440/35441：

- JWT + bcrypt + aiosqlite 完整实现
- 用户登录/注册、JWT 签发/刷新、设备管理、审计日志
- 与 PIN 体系完全隔离（不同的端口、不同的数据结构）

**主中继为 cloud_relay.py (v2 PIN-only，端口 35430)**，v3 模块保留但不作为生产默认配置。需注意的是 `requirements.txt` 中包含了 v3 的增量依赖（`bcrypt`、`PyJWT`、`aiosqlite`），这些仅 v3 模块需要。

---

## 五、Electron 端架构现状

当前 Electron 版已完成为模块化拆分：`main.js` 为 919 行编排文件，10 个独立模块（logger/settings/network/phone-notes/tray/windows/firewall/discovery/cloud/server）各司其职，采用依赖注入模式避免循环引用。`phone-connection-manager.js` 为独立模块，最大连接数 10 台（与 Go 版一致）。

## 六、部署

### 云中继

```bash
pip install websockets pystray Pillow
python cloud_relay.py
```

### PC 端

```bash
# Electron 版
cd pc-app-Electron && npm install && npm start

# Go 版
cd pc-app-go && wails build  # 输出单文件 exe
```

### 防火墙要求

- 35430 (TCP)：云中继
- 35432 (TCP)：PC 端主服务
- 35433 (UDP)：LAN 设备发现
